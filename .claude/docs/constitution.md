# Project Constitution

> 작성: 2026-06-16 | 버전: 1.0 | 최종 수정: 2026-06-16 | 상태: 초안(사용자 검토 대기)
>
> 이 문서는 프로젝트의 **불변 원칙**을 정의한다.
> 모든 설계·구현·검증 작업 전에 반드시 읽는다.
> constitution 조항이 전역 규칙과 충돌하는 경우, constitution이 우선한다.
>
> - **변경 원칙**: 조항은 합의 없이 변경하지 않는다.
> - **기준 커밋**: 문서에 commit hash·변경 이력을 기록하지 않는다. 필요 시 `git log -- <파일>` 을 사용한다.

## 1. 프로젝트 원칙

### P-001. macOS 타깃 · Bash 3.2 호환

CCTG 는 macOS 를 1차 타깃으로 하며, 셸 스크립트는 macOS 기본 `bash` 3.2 에서 동작해야 한다.

- `caffeinate`, BSD 계열 도구(예: BSD `stat`)에 의존하는 코드는 macOS 동작을 깨뜨리지 않아야 한다.
- 연관 배열 등 Bash 4+ 기능을 사용하지 않는다(메시지 카탈로그가 스칼라 변수로 작성된 이유).
- Linux/WSL 지원은 현재 목표가 아니다. (CONTRIBUTING.md 명시)

**Gates 질문**: 이 변경이 macOS / Bash 3.2 에서 동작하는가? Bash 4+ 전용 문법을 도입하지 않았는가?

---

### P-002. 전역 채널 봇·사용자 상태 비침해 (안전)

CCTG 는 전역 채널 봇과 사용자 데이터를 절대 손상시키지 않는다.

- 예약 이름(`telegram`/`discord`/`imessage`/`fakechat`)으로의 `add`/`rename` 은 거부한다(전역 채널 봇의 `.env`/`access.json` 보호).
- `rm --purge` 는 전역 채널 디렉터리·CCTG 관리 밖(비-`launch.env`) 상태 디렉터리를 삭제하지 않는다.
- `uninstall` 은 상태 디렉터리(`~/.claude/channels/`)를 보존한다.

**Gates 질문**: 이 변경이 전역 채널 봇 또는 사용자 상태 디렉터리를 덮어쓰거나 삭제할 위험이 있는가?

---

### P-003. 시크릿 비노출

봇 토큰·텔레그램 ID 등 시크릿은 프로세스 목록·저장소·로그에 노출하지 않는다.

- 토큰을 argv 로 받지 않는다(`add` 는 `--token-env`/`--token-stdin`/대화형 입력만 — 프로세스 목록 노출 방지).
- 시크릿은 상태 디렉터리 `.env` 에만 두며 저장소에 커밋하지 않는다.

**Gates 질문**: 이 변경이 토큰/ID 를 argv·커밋·로그로 노출시키는가?

---

### P-004. git/gh 변경 사용자 확인

`git add`/`commit`/`push`/`tag` 및 `gh` 발행 등 변경 동작은 실행 전 반드시 사용자 확인을 받는다.

- 본 원칙의 SoT 는 `{project}/CLAUDE.md` "Git 변경 사용자 확인 (최우선 절대 원칙)". allowlist 자동 허용을 의도적으로 덮는다.
- 파괴적 명령(`push --force`/`reset --hard`/`clean -fd`)은 대행하지 않는다.

**Gates 질문**: git/gh 변경을 사용자 확인 없이 자동 실행하려 하는가?

---

### P-005. 최소 명령 표면 · 하위호환

명령 표면은 최소로 유지하고, 레지스트리·상태 디렉터리 스키마 변경은 하위호환을 유지한다.

- churn-only 변경, 미문서화 별칭 등 표면을 늘리는 변경은 지양한다(CONTRIBUTING.md).
- 레지스트리(`projects.conf`)·`launch.env` 스키마 변경 시 기존 봇이 깨지지 않아야 한다.

**Gates 질문**: 명령/스키마 변경이 기존 등록 봇·사용자 워크플로를 깨뜨리는가? 표면을 불필요하게 늘리는가?

## 2. 예외 처리 방식

constitution 위반이 불가피한 경우 아래 절차를 따른다.

1. `plan.md`의 **"예외 사항"** 항목에 위반 조항과 근거를 명시한다.
2. 대안(위반 범위 최소화, 단계적 해소 계획 등)을 함께 기재한다.
3. 사용자의 확인을 받은 후 구현을 진행한다.
4. 구조적 위반이라면 spec 범위를 재조정하거나, 합의 후 constitution 조항 자체를 개정한다.

## 3. 품질·측정 기준 선언 (A-002)

> 파이프라인 게이트가 참조하는 프로젝트 선언. 미선언 시 게이트가 면제·임계 적용 여부를 판정할 수 없다(no-silent-caps).

- **정량 커버리지 임계**: 임계 미설정. `bats` 테스트 스위트(현 201개)를 유지·확장하되, 라인 커버리지 % 게이트는 두지 않는다. 검증 게이트는 "변경 동작에 대응하는 테스트 추가/갱신 + CI green"으로 갈음한다.
- **SLA·측정 대상 NFR 유무**: 없음 — 개발자용 로컬 CLI 도구로 SLA·성능 측정 대상 NFR 이 없다. 따라서 성능·품질 표현의 수치화를 면제한다(기능 요구의 모호 표현은 항상 구체화). 근거: `agent-rules.md §9` 조건화.
- **테스트 실행 정책 포인터**: 로컬 자동 실행 가능 여부·수동 정책은 `infra.md` §6 "로컬 개발 환경 / 테스트 실행 정책" 참조.
